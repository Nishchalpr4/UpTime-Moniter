## 💬 2. The Prompts That Shipped It (Prompt Evolution)

To guide the AI agent effectively, I evolved my instructions from simple descriptions to precise, constraint-controlled prompts:

### Backend Scaffolding

#### My first, too-vague prompt:
> "build a simple uptime monitor api in python fastapi using postgres"
*   **What came back**: A generic FastAPI backend spread across five folders (`routers/`, `schemas/`, `database.py`), using asynchronous connections that clashed with psycopg2 drivers.
*   **My refined, sharper prompt**:
    > "I need a FastAPI backend with PostgreSQL 15. Keep it inside a single main.py file to eliminate boilerplate directory nesting. Use psycopg2-binary with standard database connections. Implement the background cron loop using APScheduler in a separate thread so that blocking database calls do not freeze FastAPI's async event loop. Ensure all queries are parameterized to protect against SQL injections."
*   **What made the second prompt better**: It defined the exact database choice, isolated the scheduler running model (APScheduler background threads to prevent async event loop blocks), enforced strict single-file architecture constraints, and guaranteed SQL parameter safety.

### Frontend Dashboard Scaffolding

#### My first, too-vague prompt:
> "create a react frontend for my uptime API"
*   **What came back**: A heavy template utilizing complex canvas charts, styled in bright purple gradients that conflicted with a clean dark theme.
*   **My refined, sharper prompt**:
    > "Create a single-file React App.jsx styled with index.css. Use a desaturated dark slate layout (#0b0f19 bg, #151b2c cards) with sky blue highlights. Display cards for each URL showing current status (UP/DOWN) and average latency. Add statistic blocks at the top. Ensure adding a URL triggers an immediate API reload. Add a Refresh button that disables and displays 'Refreshing...' while fetching to prevent overlapping requests."
*   **What made the second prompt better**: Specified the exact color palette, metric layout, and defined UI-state boundaries (blocking buttons during fetches) to keep the app responsive.

## 🛠️ 3. The Course Corrections (Debugging & Refactoring)

While building, I encountered three major logic anomalies generated by the AI agent and explicitly corrected them:

### Case 1: The Async Event Loop Blocking Trap
*   **What the AI proposed**: Placing PostgreSQL database writes (`psycopg2`) directly inside an async `asyncio.sleep` cron loop in FastAPI.
*   **Why it was wrong**: `psycopg2` is a synchronous, blocking driver. Running blocking database writes directly inside FastAPI's single async event loop blocks the entire server thread, preventing incoming HTTP client requests from being processed during check cycles.
*   **My follow-up prompt to resolve it**:
    > "Do not run psycopg2 database inserts inside the main async event loop. Swapping the asyncio sleep loop with a BackgroundScheduler running on a dedicated thread. This keeps the async loop non-blocking and free to process API requests."
*   **What changed in the code**: Swapped `asyncio.sleep` with `BackgroundScheduler` in `main.py`, launching the pinger loop on an isolated thread.

### Case 2: Docker Network Host Resolution Failure
*   **What the AI proposed**: A backend `DATABASE_URL` pointing to `postgresql://uptime:secret@localhost:5432/uptime`.
*   **Why it was wrong**: Inside Docker, `localhost` resolves to the container's isolated local network loopback, not the host machine. The backend container could not resolve the separate database container on port 5432, causing connection timeouts.
*   **My follow-up prompt to resolve it**:
    > "Update the DATABASE_URL environment variable to point to the db service name in my docker-compose.yml instead of localhost."
*   **What changed in the code**: Changed the database connection string in `docker-compose.yml` to `db:5432`, letting Docker's internal DNS automatically resolve container-to-container calls.

### Case 3: The React 0ms Latency Falsy Bug
*   **What the AI proposed**: 
    ```javascript
    const lat = urls.filter(u => u.response_time).map(u => u.response_time);
    ```
*   **Why it was wrong**: In JavaScript, a number value of `0` evaluates to `false`. If a local target service responded in exactly `0ms`, this check ignored the latency, filtering it out of average calculations and showing `—` (offline) on the card dashboard.
*   **My follow-up prompt to resolve it**:
    > "Fix the response time rendering and mapping filters. A value of 0ms should not evaluate to false. Update the code to explicitly check for values that are not null."
*   **What changed in the code**: Changed the filters and displays in `App.jsx` to check `u.response_time != null`, guaranteeing that fast checks display correctly.
